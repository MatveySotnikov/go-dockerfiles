**Назначение:**

Изоляция окружения: сервис не зависит от установленного Go на хосте.

Воспроизводимость: одинаково запускается на любой машине с Docker.

Упрощённый деплой: один образ можно перенести в тестовое или прод‑окружение без дополнительной настройки.

Контрольные вопросы:
1. Чем Docker-образ отличается от контейнера?

Образ это неизменяемый шаблон, состоящий из нескольких слоёв файловой системы. В нём упакованы код приложения, зависимости, среда выполнения и метаданные (переменные окружения, команда запуска). Образ можно хранить в реестре и переносить между системами.

Контейнер это запущенный экземпляр образа. Он добавляет поверх образа собственный слой для записи, имеет изолированное окружение (процессы, сеть, файловую систему) и собственный жизненный цикл. Один образ может быть использован для создания множества контейнеров.

2. Зачем нужен Dockerfile?

Dockerfile — это текстовый файл, содержащий инструкции для автоматической сборки Docker-образа. Он описывает:

``` dockerfile
базовый образ (FROM);

установку зависимостей (RUN);

копирование исходного кода (COPY);

шаги компиляции (для Go, например);

финальную команду запуска (CMD).
```

Благодаря Dockerfile сборка становится воспроизводимой и документированной: разработчик или CI-сервер всегда получат одинаковый образ из одного и того же набора инструкций.

3. Что такое multi-stage build?

Multi-stage build (многостадийная сборка) — это техника, при которой в одном Dockerfile определяется несколько стадий с помощью нескольких инструкций FROM. 
Каждая стадия может использовать свой базовый образ и выполнять определённые задачи. Затем из предыдущей стадии копируются только нужные артефакты (например, 
скомпилированный бинарник) в финальный образ.


4. Почему для Go-сервисов удобно использовать multi-stage build?

Go компилируется в статически скомпонованный бинарный файл, который не требует внешних библиотек или рантайма (в отличие от Java, Python). Поэтому:

На первой стадии используется образ golang, содержащий компилятор и стандартную библиотеку. Здесь собирается исполняемый файл.

На второй стадии используется минимальный образ (alpine или scratch), куда копируется только готовый бинарник.

Результат — образ размером всего в несколько мегабайт, содержащий только само приложение, что идеально подходит для микросервисов и быстрой доставки.

5. Зачем нужен .dockerignore?

.dockerignore — это файл, аналогичный .gitignore, который исключает определённые файлы и папки из контекста сборки.

6. Для чего используются переменные окружения в контейнере?

Переменные окружения позволяют конфигурировать приложение во время запуска, не пересобирая образ. Это реализует принцип «конфигурация отдельно от кода».
Использование переменных окружения гарантирует, что один и тот же образ может быть запущен в различных средах (разработка, тестирование, продакшен) без изменений.

7. Что делает флаг -p при запуске контейнера?
Флаг -p (или --publish) пробрасывает сетевой порт с хоста на контейнер. Формат: `-p <порт_хоста>:<порт_контейнера>`.
Без проброса портов контейнер был бы изолирован и недоступен извне.

8. Для чего нужен Docker Compose?

Docker Compose — это инструмент для описания и управления многоконтейнерными приложениями. Вся конфигурация (сервисы, сети, тома, переменные окружения) записывается в декларативном YAML-файле (обычно docker-compose.yml).
